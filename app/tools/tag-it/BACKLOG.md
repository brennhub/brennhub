## ⏳ BRENNHUB.md 반영 대기 — 설계 정신/명언

> 이 섹션을 볼 때마다 **반드시 한 번은 `BRENNHUB.md`에 반영**할 것. 반영 완료 시 이 섹션을 **삭제**(중복·용량 방지). tag-it 추출 작업에서 정립된 정신.

- **"세상은 흑백이 아닌데 자꾸 흑백으로 자르려 했다."** 시스템이 모르는 것을 아는 척하지 않고, 불확실성을 정직하게 사용자에게 노출한다. 확정할 수 없는 것은 확정하지 않는다 — 정직한 설계다.
- **사전은 규칙을 버릴 대상이 아니라, 규칙을 발견·검증하는 토대다.** (mecab 명사 사전으로 "어느 규칙이 명사와 얼마나 충돌하는지"를 측정해 안전한 규칙만 채택한 것처럼.)
- **문맥 의존 판별(동음이의 등)은 알고리즘으로 100% 불가 = AI 영역.** AI 도입 전까지 자체 시스템을 최대한 단단히 → AI는 보완 안전망으로(호출 최소화, 비용 균형). 자체 시스템은 유사 서비스의 진입장벽을 낮추는 자산.
- **가중치·규칙은 본질적으로 "우리 데이터 안의 잠정적 추정"이지 언어의 진실이 아니다.** 보수적으로, 큰 차감 금지, 항상 조정 가능하게.

## 2차 (정리 기능 — 효능감 검증 후 투자)

- **점수 임계 슬라이더** — 점수 X 이하 칩을 실시간으로 숨김(되돌리기 가능). 기본 숨김, "이대로 삭제" 확정 버튼 별도. (기획서 §5.2 2겹)
- **올가미(rubber-band) 다중 선택 + 일괄 삭제** — 빈 영역 사각 드래그로 다중 선택. 모바일은 "선택 모드 토글 + 탭 다중선택"으로 분기. (기획서 §5.2 3겹)
- **핀(보호) UI 부활** — 보호(주황) 상태 타입·`finalTags` 로직은 코드에 유지(비파괴). MVP에선 핀 토글 UI만 숨김(보호 대상인 슬라이더·올가미 일괄삭제가 2차라 핀이 기능적으로 무의미). 위 슬라이더·올가미와 함께 칩 핀 버튼 재노출. (UI 개선 변경 1)
- **읽기 범위 확장** — 머리글·바닥글(header*.xml/footer*.xml), 각주(footnotes.xml). MVP는 본문 + 표만.

## 포맷별 추출 범위 (xlsx/pptx 확장 후속)

- **xlsx 빈도 정확화** — 현재 빈도는 `xl/sharedStrings.xml` 기준(고유 문자열 1회). 셀은 인덱스로 sharedStrings를 참조하므로 실제 셀 등장 횟수와 다를 수 있음. 시트 셀 참조(`<c><v>`)까지 세려면 무거운 처리 필요 → 효용 대비 보류(사용자가 칩 선택).
- **xlsx 범위 선택** — 시트별 선택, 차트/도형 텍스트, 정의된 이름 등. 현재 sharedStrings + 시트 inline string만.
- **pptx 범위 확장** — 발표자 노트(`ppt/notesSlides/`), 슬라이드 머리/바닥글, 도형 그룹 텍스트. 현재 `ppt/slides/slideN.xml`의 `<a:t>`만.
- **scope 포맷 적응** — `읽기 범위` 체크박스는 현재 docx 전용(표). xlsx/pptx는 본문 일괄 추출이라 무영향. 포맷별 범위 UI 분기 검토.

## 추출 품질

- **형용사 관형형 ~적인/~로운/~스러운**(0.7.1에서 처리됨) — 충돌 0~3로 안전 규칙화 완료(객관적인 류 제거).
- **바ㄹ ~인/~한 형용사 관형형** — "완벽한/같은/~인 종결" 등 충돌 大(인=1483, 한=327: 노인·법인·북한).
  Phase 3(0.8.0)에서 NNG 사전 보호가 생겼으나, **비사전 명사**(투자금 류)는 50점에서 ~인/~한
  차감 시 학살 위험 여전 → weights.ts에 추가하려면 비사전 명사 충돌까지 측정 후 보수적으로(작은
  가중치) 시도. 현재는 미추가(보류).
- **후보 양(빈도 long tail)** — 큰 문서(EBTS 25K자)는 강도5에서 ~1700 후보, 그중 62%가 1회 등장
  우발 단어. 이는 노이즈 "질"이 아니라 "양" 문제 — 세 신호(질)와 별개 축. 빈도 long tail 대책
  (강도와 분리된 가벼운 floor 옵션 / UI "1회 숨김" 토글 / 빈도순 노출 강화)은 별도 task로.
- **명사 어간 추출 반례(형용사·부사성 어근)** — "조용하였다→조용", "튼튼하게→튼튼" 등 2음절+ 비명사 어근이
  분리돼 minor noise. 드물고 net recall 이득이 커 허용. 형용사 어근 사전으로 정밀화 여지.

- **추가 언어 사전** — 일본어·중국어 등 전용 불용어 사전 추가(정밀 지원 확대). 현재 그 외 언어는 fallback.
- **핵심어 박스 코멘트** — 추출 근거(빈도·위치)를 칩 호버 시 노출.
- **추출 가중치 사용자 조정** — 빈도 vs 위치 비중 슬라이더.

## 사전 오염 — 조사 융합 표제어 (③) — (e) 적용, 나머지 보류

진단(0.8.1): mecab NNG에 조사 융합 표제어 실재 — `원의`/`원이`/`돈은` dict.has=true(100/96%).
`임이`/`뜻이`/`등에서`는 사전X지만 과도제거 가드가 통째 유지 → 50%.

**설계 검토 결론(0.8.2)** — 표면+사전조회로 융합(원의=원+의) vs 정상 한자어(회의=會議/차이=差異)
**구분 불가**. 둘 다 사전 등재 + 첫음절도 사전 명사. 알려진 정상명사 **59개 중 55개**(회의·강의·평가·
차이·종이·강도·정도·결과·효과·경로·도로·회사·의사·역사·조사·검사…)가 "어간∈사전"이라 융합 오판.

- **(a) 어간 대체+가드 완화 / (b) 사전 융합 필터 → 영구 보류.** 55/59 핵심 어휘 학살 확정.
- **(c) floor 미적용 → 무의미.** 아무것도 안 죽지만 원의도 50으로 생존(미해결) + 정상명사 100→50
  부당 하락.
- **(e) 긴 조사 가드 완화 → ✅ 채택(0.8.2).** 긴 조사(에서/으로/부터/로부터…)는 정상 명사 충돌
  ≈0이라 안전. 등에서/금액으로/회사로부터 류 해결. 단 1글자 조사(원의/돈은)는 못 건드림.
- **(d) 문서 빈도 기반 융합 병합 → 보류(조건부).** 토큰 `어간+조사`에서 어간이 같은 문서에 빈출
  후보이고 `freq(어간) ≫ freq(융합형)`이면 융합으로 보고 제거. 원의/돈은(1글자·사전O) 유일 경로.
  **단 차이/강도/회의 오제거 위험** → **반드시 실EBTS 계측 + 보수 파라미터(≥3배 등) 검증 후에만.**
  현재 보류.
  - 원의/원이/돈은(100%)은 **그냥 둔다**: 저빈도 잡음 몇 개(deselect 쉬움) < (d) 오제거 비용.
    "原義(原義)인지 원+의인지 문맥 없이 모른다 → 함부로 안 죽이고 100%로 둔다" = **확정 못 하는
    건 확정 안 한다**(brennhub 정직성)의 적용. 완전 판별은 문맥/의미 = AI 영역.

> ⚠️ **건드리지 말 것 (설계 의도)**: "있는지/삼아/곳인/흘리 = 50%"는 버그가 아니다.
> 어미 `지`(충돌 3365)·`아`(744)·`인`(1483 → 제외)·`리`(어미 아님)는 **명사 말음 충돌이 커서
> 보수적으로 w0/제외**한 것. 깎으려면 가게·녹음·노인·법인류 명사 학살을 감수해야 함. "세상은
> 흑백이 아니다" — 확정 못 하는 건 확정하지 않는다.
