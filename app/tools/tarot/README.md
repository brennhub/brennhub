# 타로 테이블 (Tarot Table) · `tarot`

변하지 않는 카드 사전으로 읽는 3장 타로 — 메이저 아르카나 22장, 전 과정 클라이언트.

> 📐 **Task 2(의식 플로우)·Task 3(리딩·저장·공유·사전 열람)의 단일 참조원.**
> 기획 원문(tarot-concept.md)은 repo 밖 — 본 문서가 §3·§5·§6 합의 사항의 정본이다.

## 한 줄 정의

> 시장 전체가 "AI가 매번 지어내는 신탁"이라면, 우리는 **"변하지 않는 사전을 정직하게 보여주는 타로"**다.
> 차별 한 문장: **"우리는 지어내지 않는다."**

결정론 카드 사전(신뢰축) + 비효율적 의식·정/역 선택(몰입축). 미래 예측은 단정하지 않는다 — 의미를 완성하는 것은 사용자다.

## 포지셔닝 & AI 정책

- **1차 출시 = 결정론 100%.** AI 호출 0, 완전 client-side, D1·외부 백엔드·외부 라이브러리 의존 0.
- AI는 기본 구축 완료 후 별도 검토. 붙이더라도 "예언자"(자세한 풀이+대화)는 기각 — 차별화 죽음. 후보는 "정리·연결 보조"(사전 내용 묶기만, 근거 노출)뿐. grounding은 환각을 *줄일 뿐 제거하지 못한다* — 명분은 "정확해서"가 아니라 "재미·편의 옵션".
- **투명성 UI 원칙**: 해석이 어느 사전 의미에서 나왔는지 화면으로 증명 (S8의 키워드↔도메인 칩 색 연결이 결정론 버전).
- **금칙**: user-facing 문구에 "AI"·가격·광고 노출 금지. "미래를 맞히지 않는다"류 문구도 대외 슬로건으로 쓰지 않는다.

## 의식 플로우 S0~S8 (와이어 정본)

단일 페이지 상태머신 — **라우트 이동 없는 단계 전환**, 모바일 세로 우선, 입구→리딩 약 2분. **느림이 곧 제품** — 비가역성이 이 도구의 심장이다. 시장 AI 타로는 다시 뽑기가 가볍다 → 우리는 컷·정/역 선택을 비가역으로 묶어 진지함을 만든다.

| 단계 | 명세 | 구현 |
|---|---|---|
| **S0 입구** | 소개 + 신뢰 한 줄("모든 해석은 공개된 사전에서 나옵니다") + [리딩 시작] + [카드 사전 열람]. 재방문 시 '지난 리딩 보기' 진입점(Task 3) | ✅ Task 1 → Task 2에서 [리딩 시작] 활성 ([사전 열람]은 Task 3) |
| **S1 그라운딩** | 어두운 전환, 느린 펄스 비주얼 + "마음속 질문을 하나로 좁혀보세요" 1줄. **약 6~8초 고정, 스킵 없음, 진행바 금지**(로딩처럼 보이면 실패) — 호흡 리듬 연출 | ✅ Task 2 |
| **S2 질문** | 질문 직접 적기(textarea, **빈 질문 진행 불가**, "리딩 화면에 그대로 표시됩니다" 안내 = 정성 유도) + 도메인 칩 **단일 선택**(5종) + 마이크로카피 "질문은 이 기기 밖으로 나가지 않습니다". 질문 텍스트는 **어떤 네트워크 요청에도 포함 금지** | ✅ Task 2 |
| **S3 셔플** | **자동 애니메이션 금지** — 드래그/스와이프로 직접 섞기. 최소 3회 인터랙션 후 [이제 됐어요] 버튼 등장(끝내는 시점도 사용자 결정). 엔트로피: `crypto.getRandomValues` + 인터랙션 좌표·타이밍 시드. 점퍼 카드 연출 자리(v1.5) | ✅ Task 2 |
| **S4 컷** | 두 지점 탭/드래그로 3더미 분할 → 합칠 순서 탭 선택(**선택 순서가 실제 덱 순열을 결정** = 정직한 인터랙션). 합치는 순간 **카드 순서 확정(봉인)** + "이제 되돌릴 수 없습니다" 연출 + 봉인 해시 한 줄 표시. **이후 이전 단계 복귀 불가 — 수정하려면 처음부터** | ✅ Task 2 |
| **S5 스프레드 선택** | 봉인된 덱 22장을 **뒷면**으로 펼침 — 좌→우 = 봉인 덱 순서 0→21, 추가 난수 없음. 3장을 차례로 탭 → 1번째=과거, 2번째=현재, 3번째=미래 자리로 이동(빈자리 유지, 재배열 없음). **탭 = 확정, 취소 없음.** `pickedIndices` 기록(Task 3 검증 표시용). 3장 후 0.8초 → S6. 봉인이 전체 덱 순서를 덮으므로 임의 위치 선택도 커밋-리빌 무결. 배치가 선택(S6)보다 앞 — 눈앞에 놓인 카드에 운명을 거는 긴장감 | ✅ Task 2 → 2.5 교체 |
| **S6 정/역 선택 ★** | 단층 — 아래 "정/역 선택 (단층)" 참조. 문구 고정: **"세 장을 어느 방향으로 뒤집을지, 당신이 정합니다. 되돌릴 수 없습니다."** [정방향으로 뒤집기]/[역방향으로 뒤집기]. **꾹 눌러 약 1.5초 홀드 확정**(진행 링 표시) — 의도적 마찰. **더블 확인 모달 금지**(긴장 희석) | ✅ Task 2 → 2.5 단층 정정 |
| **S7 오픈** | 사용자가 각 카드를 **직접 탭해 뒤집음**, **과거→현재→미래 순서 강제**. 뒤집힘 = 카드 앞면 + 이름 + 방향 뱃지 | ✅ Task 2 |
| **S8 리딩** | 상단: 적은 질문 원문(따옴표) + 도메인 뱃지 = 대가의 회수(질문이 리딩의 제목). 카드별 섹션: essence(항상) → 도메인 매칭 키워드 강조 + gloss(**도메인 칩과 동일 색으로 시각 연결**) → 비매칭 키워드 접힘('전체 키워드 보기' 토글 — 숨기지 않되 강조만) → 매칭 0개면 mute 문구 → [원전 보기] 토글(Waite 영문 원문) → '검증' 접힌 토글(커밋-리빌 원본 공개). 하단: [이미지로 공유](질문 미포함 기본) / [새 리딩](S0부터 — **같은 리딩 재뽑기 없음**, 전통 "같은 질문 두 번 묻지 않기") / [사전 열람] | Task 3 |

**대가 의식**: 질문 직접 적기로 번안 — 돈(복채) 대신 정성·시간이라는 대가. Task 2는 오픈 완료 시 임시 결과(카드 id + 방향 텍스트 나열)까지만 — S8 렌더링은 Task 3.

## 정/역 선택 (단층)

S6에서 고른 방향이 **세 장 전체의 방향을 직접 결정**한다 — 역방향 선택 = 3장 모두 역방향, 정방향 선택 = 3장 모두 정방향. 카드 방향의 단일 소스는 `userChoice`(숨은 상태 없음).

선택의 무게는 메커니즘 복잡성이 아니라 **비가역**에서 나온다 — 홀드 1.5초 확정, 모달 없음, 선택 후 복귀 불가. 역할 분담: **어떤 카드**가 나올지는 봉인된 덱 순서(운명)가, **어느 방향**으로 볼지는 사용자(책임)가 정한다.

> 정정 이력: 2026-06-11 편집장 결정으로 2층(숨은 비트 × 선택 XOR) → 단층. dev 실물 검수 확정.

## 커밋-리빌 검증 (MVP 포함)

증명 대상 = **"카드를 고르기 전에 덱 순서가 고정되어 있었다"** — 말 아닌 시스템 강제. 방향은 사용자의 투명한 선택이라 증명 대상이 아니다.

- S4 확정 시: `SHA-256(카드 순서 + nonce)` 봉인 해시 한 줄 표시 (Web Crypto, client-side, 외부 의존 0).
- S8: 접힌 '검증' 토글에서 원본(순서 + nonce) 공개 → 사용자가 해시 재계산으로 검증 가능.
- **일정 압박 시 1순위 컷 후보** (BACKLOG 명기, 미발동).

## 카드 사전

- **범위**: 메이저 아르카나 22장 × 정/역 = 44 항목. 마이너 56장은 v2.
- **스키마** (`lib/tarot/types.ts` — 합의 2026-06-10): 카드 의미는 **포지션 비의존** — 과거/현재/미래는 표시 라벨일 뿐. 텍스트 필드는 `{ ko, en? }`.

```ts
type Domain = "love" | "work" | "money" | "relation" | "self";
// 칩 라벨: 연애 / 일·직업 / 돈 / 관계 / 자기 — i18n tarot.domain_*, enum 값은 내부 고정

interface TarotCard {
  id: number;                 // 0~21
  slug: string;               // "strength", "death", "high-priestess" ...
  name: { ko: string; en: string };
  meta: { numerology: number; element?: { ko: string }; astrology?: { ko: string }; symbols: { ko: string }[] };
  upright: OrientationEntry;
  reversed: OrientationEntry;
}
interface OrientationEntry {
  essence: { ko: string; en?: string };   // 2~3문장 핵심 의미 — 항상 노출
  waite: string;                          // 1910 원전 점술 의미 원문(영문)
  keywords: { word: {ko; en?}; domains: Domain[]; gloss: {ko; en?} }[];  // 방향당 4~6개
}
```

- **리딩 로직**: 선택 도메인 ∈ `keyword.domains` → 해당 키워드 강조 + gloss 노출. essence는 항상 노출.
- **mute 정직성 장치**: 매칭 키워드 0개면 억지 해석 금지 → 정확 문구 **"이 카드는 [도메인]에 직접 닿는 키워드가 없어요. 전체 의미로 보세요."** + essence만. (현재 데이터 기준 mute 조합 9건 — gen-cards.mjs 로그 참조)
- **질문 처리**: 자유 질문은 의식·대가·표시용 — **텍스트 분석 안 함. 가짜 NLP 금지** — 질문을 이해하는 척하지 않는다. 결정론 매칭 입력은 도메인 칩뿐.
- **출처 3층 구조** (혼동 금지): ① `waite` 필드 = 1910 *The Pictorial Key to the Tarot* 원문 인용 — 검증 완료, 오류 무관용 ② 점성·원소 = 골든던 전통(원전 책에 없음 — 원전 출처로 표기 금지) ③ essence·키워드·gloss = 현대 통용 합의 기반 자체 큐레이션.

## 프라이버시

- 적은 질문은 **기기 밖 미전송** (localStorage만). 리딩 결과 서버 저장 없음(MVP).
- 공유는 이미지 카드만 — 질문 텍스트 기본 미포함.
- 공유 이미지는 `canvas.toDataURL()` 직접 그리기 — **html2canvas 금지** (Tailwind v4 + Lightning CSS에서 빈 PNG, lineup-builder에서 확인된 제약).

- 봉인 payload 정본: `tarot-seal-v1|order:<22 ids>|nonce:<32hex>` → SHA-256. **포맷 변경 금지** — S8 검증 토글(Task 3)이 재해시. (2026-06-11 단층 정정으로 bits 필드 제거 — prod 미출시라 마이그레이션 없음)
- 새로고침 = 의식 리셋 (메모리 상태만 — 영속은 Task 3 범위).

- `?debug=1` — S0 하단에 22장 앞면 그리드 (편집장 dev 검수·양 테마 시각 검증용). S5~S7 구현 시 제거 가능.
- 카드 비주얼: 선출시 = CSS 타이포 카드 (제작 병목 0, 의식 무드 통일). RWS 1909 도판 스캔은 앞면 비채택 — 전 경쟁 앱과 동일 비주얼 = 차별 죽음. 픽셀아트 덱은 v1.x 교체 트랙.

## i18n 정책

- UI 크롬(버튼·라벨·안내·의식 문구)은 **ko/en 필수**. 사전 본문(essence·키워드·gloss)은 **ko 전용 출시** — 의도된 예외 (본문 톤은 번역이 아니라 재집필 + 영어권 타로 SEO 효익 없음, saju-naming 선례). en locale에서 카드 이름은 영문 데이터, 본문은 ko 노출.
- 도메인 칩 라벨: i18n `tarot.domain_love/work/money/relation/self` (supp-plan `cat_${c}` 동적 룩업 패턴).

## 상태 저장 (Task 3 가이드)

마지막 리딩 1건 localStorage — **PATTERNS.md의 hydrate/persist/schemaVersion 패턴 그대로** (stock-sim·tag-it 선례). 재방문 시 S0에 '지난 리딩 보기' 진입점. 질문 포함 저장 — 기기 내에만. Task 2는 메모리 상태만(영속 없음).
